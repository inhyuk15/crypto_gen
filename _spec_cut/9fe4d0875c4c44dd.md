# **Specification** 

SHELL is a block-cipher-based authenticated encryption mode. We recommend AES [1] as the underlying block cipher of SHELL, since it is the most widely used block cipher nowadays, The instantiation of SHELL with AES is denoted as SHELL-AES in this report. 

#### **1.1 Parameters** 

SHELL uses the following parameters. 

- **Block cipher** _E_ : _{_ 0 _,_ 1 _}_<sup>_k_</sup> _× {_ 0 _,_ 1 _}_<sup>_n_</sup> _→{_ 0 _,_ 1 _}_<sup>_n_</sup> . 

- **A set of auxiliary permutations** _{APi}_ : _APi_ : _{_ 0 _,_ 1 _}_<sup>_ki_</sup> _× {_ 0 _,_ 1 _}_<sup>_n_</sup> _→ {_ 0 _,_ 1 _}_<sup>_n_</sup> . Note that _ki_ is not necessarily equal to _k_ . 

- **Key length:** _k_ bits or 2 _k_ bits. 

- **Nonce length** _ℓ_ nonce **:** a positive integer such that 1 _≤ ℓ_ nonce _< n_ . 

- **Tag length** _ℓ_ tag: a positive integer such that 1 _≤ ℓ_ tag _≤ n_ . 

- **Frame width** _w_ **:** a positive integer. 

- **Maximum plaintext block length** _ℓ_ pt **:** _ℓ_ pt _≤_ 2<sup>_n−ℓ_nonce</sup><sup>_−_1</sup> . 

- **Maximum number of plaintexts under the same key:** 2<sup>_ℓ_nonce</sup> . 

- **The number of auxiliary permutations** _d_ **:** a positive integer. 

#### **1.2 Recommended parameter sets** 

We recommend the following as the default parameters. 

- **Block cipher** _E_ **:** AES-128. Thus _k_ = 128 and _n_ = 128.

- **A set of auxiliary permutations** _{APi}_ : all 4-round AES permutations with independent round keys. 

- **Key length:** 128 bits. 

- **Nonce length:** _ℓ_ nonce = 64 or 80. 

- **Tag length:** _ℓ_ tag = 128. 

- **Frame width:** _w_ = 256. 

- # **auxiliary permutations:** 4 _≤ d ≤_ 8. 

#### **1.3 Authenticated encryption mode SHELL** 

As an authenticated encryption mode, the encryption algorithm _EK_ of SHELL takes a nonce _N_ , an associated data _A_ and a plaintext _M_ as input, and produces a ciphertext _C_ and a tag _T_ as output. That is ( _C, T_ ) _←EK_ ( _N, A, M_ ). The decryption algorithm _DK_ of SHELL takes a nonce _N_ , an associated data _A_ , a ciphertext _C_ and a tag _T_ as input, and outputs either a plaintext _M_ if the tag is valid, or _⊥_ otherwise. That is _M/⊥←DK_ ( _N, A, C, T_ ).

##### **1.3.2 Structure overview of SHELL** 

We present a high-level overview of the structure of SHELL, which is also shown in Figure 1.1. We will omit the description of the decryption procedure since it can be trivially obtained from the encryption procedure.

The nonce _N_ is input to a Pseudo-Random-Permutation (PRP) layer CENC [10], which maps it to pseudo-random strings that are indistinguishable from ideal random strings with a provable security beyond the birthday bound. Then, _V_ and _I_ are input to an encryption layer called PX-Enc, which shares the same basic primitives with PX-MAC and thus is also universal-hash-based. Finally, _F_ , the checksum of _I_ and _U_ are input to tag generation, and a tag _T_ is produced. 

The above structure overview also well explains why we named it SHELL as shown in Figure 1.2. Our design consists of three layers. The upper and the lower layers utilize strong cryptographic primitive pseudo-random-permutations, e.g., full-round AES. The middle layer utilizes much weaker cryptographic primitive differentially-uniform permutations, e.g., four-round AES with independent round keys. 

Moreover, we would like to regard the universal hash layer, more precisely PX-MAC and PX-Enc, as the main novelty of this design (to our best knowledge), or in other words the _meat_ of SHELL.

# **Security goals** 

SHELL uses the public message number as nonce. The secret message number has length of 0 bits. 

The security goal of SHELL is twofold. On one hand, in the nonce-respecting environment where no nonce is repeated to the encryption algorithm, we expect SHELL can provide a provable security beyond the birthday bound. On the other hand, in the nonce-misuse environment where the same nonce can be used for distinct queries to the encryption algorithm, we expect SHELL can still provide a provable security bound, which we choose to be slightly lower than the birthday bound. This is because a higher security bound usually comes with an efficiency loss. We have to take the tradeoff into consideration. 

The security goal of SHELL mode is provided in Table 2.1. The definitions of security notions are detailed in Section 3.2. 

It is important to note that to our best knowledge, the privacy security bound naturally implies (at least) the same security bound for the confidentiality of the plaintext, and the authenticity security bound naturally implies (at least) the same bound for the integrity of nonce and the integrity of associated data.
