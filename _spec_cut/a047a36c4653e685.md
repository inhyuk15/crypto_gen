## **3 Context separation** 

From an application perspective, new AEGIS variants should ideally share the same interface as existing variants. Namely, they should accept a single message, optional associated data, a 128- or 256-bit key, and a 128- or 256-bit initialization vector. 

However, AEGIS is meant to be used in a nonce-respecting scenario: Vaudenay and Vizár [VV18] analyzed the robustness of CAESAR candidates, including AEGIS, and emphasized that schemes like AEGIS critically require nonce-respecting usage to maintain their security guarantees. Their work highlights that nonce reuse can lead to catastrophic security failures in many AEAD schemes. Clearly, reusing the same key and _IV_ to encrypt different parts of a message would violate this requirement. 

In order to avoid universal forgery and decryption attacks as identified by Vaudenay and Vizár, we augment the AEGIS initialization functions with a context to provide domain separation. This is essential to ensure that each parallel fragment effectively operates as a separate cryptographic instance, preventing the security issues that would arise from direct key and IV reuse. That is, for two different values _CTX_ 0 = _CTX_ 1, _Enc·_ [ _·_ ]( _CTX_ 0 _, ·_ ) and _Enc·_ [ _·_ ]( _CTX_ 1 _, ·_ ) act as two different functions of _{K, IV, A, P }_ . 

This approach is conceptually similar to the domain separation used in Keyak’s "river" (parallel) mode [BDP<sup>+</sup> 16], where the lane index is encoded into the state to ensure that each parallel instance processes data independently despite sharing the same key. Keyak’s design explicitly notes that when running with multiple parallel lanes, keystreams are domain-separated by encoding the lane index, similar to our context byte approach. 

A context is made of two bytes: a byte that provides separation between parallel instances, and another byte representing the parallelism degree. 

The context acts as a mask applied to specific words of the AEGIS states during initialization.

## **6 Implications of the AEGIS-128X context separator** 

The context parameter is a small, public tag that we XOR into state words _S_ 3 and _S_ 7 at the start of every initialisation round. This section walks through why a single-bit change in that parameter spreads through the whole 1024-bit state within a handful of rounds and why an attacker cannot undo that effect by choosing clever nonces. Throughout, we use the usual _ideal-cipher model_ : every call to `AESENC` (which implements a single round of AES encryption) acts as an independent, random permutation of 128 bits.

## **9 Implementation notes** 

Implementing AEGIS-128X[ _ν_ ] and AEGIS-256X[ _ν_ ] only requires trivial modifications to existing AEGIS-128L and AEGIS-256 implementations. 

They apply the exact same operations as AEGIS-128L and AEGIS-256 to vectors of _ν_ AES blocks instead of single blocks.

On CPUs that don’t implement vectorized versions of the AES permutation, AEGIS-128X[ _ν_ ] and AEGIS-256X[ _ν_ ] can be implemented in different ways: 

- by emulating AES vector instructions. This is the easiest option, keeping the code close to hardware-accelerated versions. 

- by evaluating _{A_ 0 _, A_ 1 _, A_ 2 _, . . . Aν−_ 1 _}_ and _{C_ 0 _, C_ 1 _, C_ 2 _, . . . C ν−_ 1 _}_ sequentially, with a periodic synchronization, for example after every memory page. This reduces cache-locality but also register pressure. 

## **11 Conclusion** 

We propose two additions to the AEGIS family of authenticated encryption algorithms: AEGIS128X[ _ν_ ] and AEGIS-256X[ _ν_ ]. 

They take advantage of new vector instructions to compute multiple instances of AEGIS-128L and AEGIS-256 in parallel. In order to do so, inputs are simply split into _ν_ pieces that can be processed independently, in a way that directly matches how CPU instructions load, store and use vector registers. 

As a result, they are very similar to the non-parallel variants, share the same security guarantees, yet significantly improve the performance of these ciphers on CPUs supporting these vector instructions.
